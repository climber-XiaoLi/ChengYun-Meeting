# 澄云会议

------



## 项目概述

一款主要依赖C++语言，基于C/S架构开发的实时音视频会议项目，可实现用户注册/登录，短信验证注册，会议房间创建/加入，实时音视频通信、屏幕共享，人脸识别特效，音频降噪（静音检测）。

## 设计

### 客户端：

基于Qt框架开发，结合QThread 多线程 + 信号-槽机制解耦业务逻辑与 UI 渲染，保证界面高流畅性；采用多 TCP 连接分离信令与音视频数据流，避免互相阻塞导致的通信延迟；集成Speex 音频压缩 + H.264 视频编解码，大幅降低带宽占用；接入WebRTC-VAD 静音检测模块，有效过滤环境杂音；集成OpenCV + 轻量级 CNN 人脸检测模型，实现人脸实时追踪与萌拍特效动态跟随；使用MD5 加密用户密码，保障账号安全；通过SQLite本地存储用户信息、会议历史等数据；运用正则表达式校验用户输入合法性；采用单例、中介者等设计模式提升代码可维护性与扩展性。

### 服务端：

采用epoll + 线程池高并发模型，结合边缘触发 + 非阻塞 Socket + 状态机架构，支撑万级并发用户连接；使用MySQL持久化存储用户信息、会议房间等全局数据；应用层自定义结构体协议，拆包处理客户端请求。

## 技术点

### MySQL

**概念：**开源关系型数据库，适合服务器大规模数据存储与关联查询。数据安全性高，并发读写能力强。

**作用：**服务端需要存储用户信息（账号、密码、昵称)，会议房间状态、操作日志等全局数据。要支持多客户端同时访问和数据共享，同时需要保证数据读写的安全性和并发稳定性，MySQL的事务机制和并发处理能力，完全适配服务端全局数据持久化的需求

**实现：**

用户信息表（存储加密后的账号密码、手机号）

会议房间表（记录房间号、创建人、在线成员、房间状态）

操作日志表（记录用户登录、房间创建销毁等行为）

**服务端封装数据库增删改查接口**，完成用户注册信息存储、登录验证、房间状态同步等操作。

### 视频编解码：H.264

原始视频帧数据量极大（一帧摄像头YUV格式图像(640x480x12)450KB  → 4.5KB）（例如 30 帧 / 秒的视频，原始数据每秒可达数十MB）尤其是桌面采集将会更高。

H.264 的**高压缩比**（压缩比可达 100:1 甚至更高）能在保证视频清晰度的前提下，将视频数据体积压缩至原始的 1%~10%，**降低网络带宽压力**。且可以使用低延迟模式，**编码延迟极低**，完全适配实时音视频的低延迟需求。

**最终效果：**

平衡编码速度与压缩效果。压缩比达到 100：1左右。编码延迟：10ms以内

视频：640 * 480 * 12 * 15 =  110592kbps      H264压缩（100 : 1）---> 	**550kpbs**

桌面:  1280 * 800 * 12 * 15 = 368640kpbs	H264压缩（100 : 1）---> 	**1843kpbs**

桌面共享，主要用来讲PPT，帧率可以设置的更小一些。

### 音频编解码：Speex音频压缩

#### **设计实现：** 

**为了适配更多麦克风**，8000Hz采样率+单声道+16位，换算后每20ms采集320字节；一秒50次；

而speex编码器正好要求每次编码输入为320字节；

使用定时器，每20ms处理一次音频缓冲区的数据采集，若数据大于640字节进行一次压缩并发送。因为speex要求每次640字节压缩。

**最终效果：**

一帧20ms时间内，采集的原始音频数据640字节 ----speex压缩后---> 数据量是76字节；

压缩可比 8.4 : 1 

音频的码率为15-16kbps

### 服务端epoll+线程池

#### **作用：**

项目支持**多客户端同时加入会议、发起音视频请求**，服务端需要处理**高并发请求**。epoll负责监听所有客户端连接和数据请求，线程池负责处理具体业务逻辑，兼顾高并发处理能力和低资源消耗。

#### **实现：**

epoll监听客户端的连接请求、数据发送；**生产者线程**将对应的业务放入**任务队列（双向环形队列）**；**线程池使用消费者生产者**，使用**互斥锁、条件变量**实现线程同步，**消费者线程**从队列接去任务执行（接受连接、接收数据、处理），**管理者线程**负责监控任务量与线程数，扩容和缩减线程数量。

### OpenCV人脸检测强化版-萌拍特效

#### **概念：**

libfacedetection 是由俞士祺团队开源的高性能**轻量级**人脸检测库， 底层基于 **深度卷积神经网络**（CNN）yunet模型优化，专门**针对 CPU*实时推理做了加速，精准度：>95%**。

#### **作用：**

萌拍特效需要实时跟随人脸移动，必须保证 25~30fps 的检测帧率，防止**特效跟不上人脸移动**，libfacedetection 纯 CPU 推理在640x480 分辨率下能轻松跑满 30fps。

#### **实现：**

下载源码动态库，**调用其中核心函数接口检测**；对摄像头采集画面进行人脸检测，**返回脸部矩形框坐标**，依据这个坐标将特效图片绘制在居中位置；将视频采集和人脸识别工作**放入客户端的多线程中执行**，防止主界面操作卡顿。

#### **对比：**

一开始使用的OpenCV的**Haar算法**，是传统机器学习算法，基于**人工特征**（边缘、线性、中心特征），先**检测脸部**，再在脸部范围内**检测眼睛**，判断是否是人脸来丢弃。效果不好（手挡一下脸 ，眼镜遮挡，导致跟丢）

深度学习模型，兔耳朵、圣诞帽可以**精准、流畅、稳定地**跟随人脸;且可筛选**置信率80%**以上的结果，不会错误识别。

### WebRTC-vad 静音检测

#### **概念理解 + 底层原理：**

WebRTC VAD 是 Google WebRTC 内置的、专为语音通话设计的**语音活性检测模块**，属于业界工业级标准的静音检测方案。

处理音频的原始 PCM 数据；s

提取人声特征，判断当前帧是**人声**还是**静音/噪声**；能区分人声、风声、电流声。

输出 4 种状态：0=静音，1=低能量语音，2=中能量，3=高能量；

#### **作用：** 

**节省带宽：**用户不说话时，不发送音频数据；

**提升通话体验：**避免环境噪音持续传输，让会议更安静。

#### **实现：**

下载源码动态库；要求采样率（16kHz）、帧长度（10ms/帧），但我**为了适配更多麦克风**，8kHz，每20ms处理一次音频缓冲区的数据，每320字节，调用WebRTCVad接口进行静音检测；若检测结果为静音 → **不编码、不发送音频**。

### 客户端多线程

#### **概念与底层：**

我的项目客户端多线程，是基于 Qt 的 **QThread** 结合 **信号与槽** 机制实现的。 采用 **moveToThread** 的方式将业务逻辑移到子线程，通过信号槽做线程间通信。

**QueuedConnection（队列连接）机制：**主线程发出信号；Qt 底层把**信号 + 参数**打包，放进**目标线程的事件队列**，立即返回，不等待执行；目标线程事件循环取出执行**槽函数在目标线程里跑**；子线程检测完成，发出信号（携带检测好的图像）给主线程，再网络发送。

Qt 会自动做**深拷贝**，给目标线程一份**独立副本**，所以**完全无锁，线程安全**。

**实现：** 音视频采集，WebRTC VAD 静音检测，人脸检测 ，音频编码、H.264 视频编码，网络发送 放入子线程。UI 主线程只负责界面显示、按钮点击、画面渲染。

**作用：**可以让耗时任务**不阻塞主线程**；

子线程异常不会导致主线程崩溃。

### MD5加密

**概念：**MD5是单向哈希加密算法，底层通过对原始数据进行哈希运算，生成**固定128位的哈希摘要**，不可逆运算。

**为什么用：**敏感数据暴漏在网络中会被不法之人劫取，保障用户账号安全。

**实现：**客户端先将**密码明文+固定盐值**加进行MD5加密，生成**哈希摘要**后发送至服务端，服务端将摘要**存储至MySQL数据库**；用户登录时，客户端再次加密密码，将摘要发送至服务端，服务端对比数据库存储的摘要，一致则验证通过，**全程不传输和存储明文密码。**

**优缺点：**加密速度快，**实现简单**，满足基础密码安全需求；缺点：安全性中等，存在**暴力破解和彩虹表破解**风险，后续考虑换成对称通信+非对称加密对称密钥的混合加密方式

## 优化

### 人脸检测：

普通：一开始使用的OpenCV的**Haar算法**，是传统机器学习算法，基于**人工特征**（边缘、线性、中心特征），先**检测脸部**，再在脸部范围内**检测眼睛**，判断是否是人脸，来丢弃误识别的对象框。效果不好（手挡一下脸 ，眼镜遮挡，导致跟丢）

改进：最后使用轻量级的深度学习模型，效果更好，满足了，精度和检测速度的平衡需求。实现效果几乎完美的特效跟踪！

### 服务端的多路监听模式：

普通：水平触发LT + ONESHOT + 阻塞socket

改进：边缘触发ET + 非阻塞socket +状态机

### 发送客户端：

1、采用时间戳丢包的方式，降低发送缓冲区压力，以及延迟影响。

2、音视频压缩，降低带宽，提高发送速度。

3、Qthread多线程+队列链接机制，防止用户UI界面卡顿。

4、H264参数上的改进，降低延迟，平衡压缩效率。
